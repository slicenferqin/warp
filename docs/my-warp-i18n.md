# My Warp 汉化与 Fork 维护规划

本文档用于约束 `my-warp` 后续汉化与 fork 维护工作。它只描述项目定位、技术路线、品牌边界、实施范围和验收标准，不代表本阶段已经完成运行时代码改造。

## 项目定位

`my-warp` 是基于官方 Warp 开源仓库维护的个人化 fork，用户可见产品名使用 `My Warp`。

这个 fork 的第一阶段目标是中文本地化，让中文系统用户能在主要产品路径中看到一致、可维护、可切换的中文界面。后续可以在同一 fork 中继续扩展隐私默认值、AI UI 弱化、默认体验调整和自用打包分发，但这些方向不进入第一阶段的汉化验收范围。

`my-warp` 不是官方 Warp，不应以官方 Warp 的身份对外发布、支持或声明兼容承诺。`my-warp` 也不是 `Warp Pro`、`Warp Plus` 或任何形式的破解版，不应在命名、文案、打包元数据或发布说明中暗示绕过官方授权、订阅、服务端能力或商业限制。

## 命名与品牌边界

- 仓库名使用 `my-warp`。
- 用户可见产品名使用 `My Warp`。
- 代码包、crate、模块和配置名可以按 Rust 与仓库约定使用 `my_warp` 或 `warp_*` 前缀，但新建的本地化抽象应避免和官方服务端 API、官方品牌资产产生混淆。
- 不使用 `warp-pro`、`warp-plus`、`oh-my-warp`、`Warp Pro`、`Warp Plus` 作为公开项目名、二进制名、bundle 名或发布名称。
- 保留官方 Warp 上游版权、许可证和必要署名。当前仓库中 `warpui_core` 与 `warpui` 使用 MIT license，其余代码使用 AGPL v3；fork 发布和源码分发需要继续遵守这些许可证边界。

## 与 warp-cn 的关系

`warp-cn` 可以作为参考实现和部分翻译素材来源，但不作为 `my-warp` 的主线 fork，也不直接合并其完整 diff。

后续实现者可以阅读 `Heartcoolman/warp-cn` 来理解已有中文化尝试、翻译术语和工程落点，但迁移时必须逐项评估：

- 是否符合 `my-warp` 的 `warp_i18n` API 边界。
- 是否会引入和当前上游代码结构不匹配的改动。
- 是否包含品牌、破解、订阅绕过或与本 fork 定位不一致的内容。
- 是否能通过本文件定义的 key parity、构建和手动验收。

如果复用 `warp-cn` 的代码、翻译条目或明显源自其实现的结构，需要在对应 PR、变更记录或翻译文件注释中保留来源说明：

```text
Translation/code reference: Heartcoolman/warp-cn, commit <commit-sha>
```

未复用其具体内容、仅作为调研参考时，也建议在 PR 描述中说明参考范围，避免后续维护者误判来源。

## i18n 技术路线

第一版消息格式采用 Fluent / FTL。选择 FTL 是为了在早期获得成熟的本地化资源组织、变量插值、选择逻辑和翻译工具生态，同时避免在产品代码中直接散落底层 FTL 解析细节。

运行时代码不应直接依赖 FTL 文件布局或具体解析库。后续应新增 `crates/warp_i18n` 作为唯一的本地化入口，通过自有 API 隔离底层实现，保留未来迁移到 MessageFormat 2 或其他消息格式的空间。

建议的最小接口如下：

```rust
t!("settings.language.title")
tr(dynamic_key)
```

其中：

- `t!("key")` 用于静态 key 翻译，后续应尽量支持编译期或构建期校验。
- `tr(key)` 用于少数运行时 key 场景，必须限制使用范围，并由 i18n 校验任务覆盖缺失 key。
- `Locale` 至少支持 `zh-CN` 和 `en`。
- `LanguagePreference` 至少支持 `ZhCn`、`En` 和 `System`。

语言资源建议按以下布局组织：

```text
crates/warp_i18n/
  bundles/
    en/
      *.ftl
    zh-CN/
      *.ftl
```

FTL 文件应按产品区域拆分，而不是按开发者个人习惯任意拆分。第一阶段可以从 Settings、菜单、按钮、登录启动路径和命令面板开始；后续再按功能域扩展。

## 设置与默认语言策略

后续实现时，Settings 应新增一个顶层语言设置：

```toml
language = "zh-CN" | "en" | "system"
```

默认值为 `system`。

语言解析策略：

- `system`：中文系统环境显示中文，其他系统环境显示英文。
- `zh-CN`：强制显示简体中文。
- `en`：强制显示英文。

语言切换应尽量即时生效。Settings 导航、常见按钮和产品内菜单应在切换后更新；macOS 顶部菜单如果无法局部刷新，应在语言切换后重建菜单。

## 第一阶段范围

第一阶段以高频、低风险、用户能立刻感知的界面为主：

- Settings 顶层导航和常见设置项标题。
- 应用菜单、上下文菜单和 macOS 顶部菜单的高频项目。
- 通用按钮，例如 OK、Cancel、Apply、Save、Close、Back、Next、Retry、Delete、Edit。
- 启动、登录、登出和基础账号状态提示。
- 常见错误提示、toast、空状态和确认弹窗。
- 命令面板高频项，包括打开设置、切换主题、打开命令、窗口与会话常用操作。

第一阶段不追求完整覆盖所有页面。目标是建立稳定的 key 体系、加载机制、语言偏好设置和质量检查，让后续页面迁移可以持续进行。

## 暂缓范围

以下内容暂缓进入第一阶段，不应为了追求覆盖率而强行改写：

- AI / Agent 深层页面和复杂运行态消息。
- 服务端返回内容，包括远端错误、云同步对象、公告、计费或账号服务文案。
- AI 模型回复、工具调用结果和 agent 生成内容。
- 终端命令输出、shell 提示、PTY 内容和用户项目文件内容。
- CLI 文案和外部工具输出。
- 上游官方文档、官网链接、许可证正文和第三方依赖许可证内容。

本地翻译层只负责客户端 UI 字符串，不应改写用户数据、模型输出、服务端协议内容或终端内容。

## 翻译规范

中文翻译应稳定、短句优先，避免机器翻译腔。术语应在翻译文件或后续术语表中统一维护。

第一阶段建议统一以下术语：

| English | 中文 |
| --- | --- |
| Settings | 设置 |
| Command Palette | 命令面板 |
| Workspace | 工作区 |
| Session | 会话 |
| Terminal | 终端 |
| Agent | Agent |
| Drive | Drive |
| Login / Sign in | 登录 |
| Logout / Sign out | 退出登录 |
| Sync | 同步 |
| Theme | 主题 |
| Keybinding | 快捷键 |

专有名词、协议名、模型名、命令名、文件名和用户可复制的代码内容默认不翻译。`Agent`、`Drive` 等 Warp 产品内概念在第一阶段可以保留英文，以减少和普通中文词汇的歧义；后续如要翻译，需要在术语表中统一变更。

## 实现约束

后续代码实现应满足以下约束：

- 不新增裸 UI 字符串。新增用户可见文案必须通过 `warp_i18n` key 进入资源文件。
- 英文资源是 key parity 的基准，中文资源必须提供同名 key。
- 静态 key 优先使用 `t!("key")`，动态 key 只能在运行时枚举确实不可避免时使用 `tr(key)`。
- 产品代码不直接读取 FTL 文件，不直接持有 Fluent bundle，不直接拼接本地化文件路径。
- 文案插值必须使用结构化变量，不使用 `format!` 拼接半句翻译。
- 复数、选择和平台差异文案应放在消息格式中处理，不在 UI 代码中分叉出多套自然语言句子。
- 翻译缺失在开发构建中应明显失败或报警；发布构建至少要回退到英文，避免显示空字符串。
- 语言偏好属于用户设置，应随现有设置持久化机制保存。
- 不通过本地翻译层改写服务端响应、终端输出、模型回复或用户内容。

## 质量门槛

第一阶段实现完成后，至少需要满足：

- 中英文 key parity：`en` 与 `zh-CN` 的 key 集合一致。
- 无裸 UI 字符串新增：新增可见文案能被 i18n 检查发现。
- 语言切换可见：Settings、菜单和常见按钮在切换后能看到语言变化。
- 中文术语统一：同一产品概念不在不同页面出现多种译法。
- `warp-oss` 可以构建。

建议的自动化验收命令：

```bash
cargo check -p warp_i18n
cargo check --bin warp-oss -p warp
cargo xtask check-i18n --check-parity
cargo xtask check-i18n --mode hard
```

其中 `cargo xtask check-i18n` 是后续需要新增或接入的校验入口；在该命令落地前，PR 需要说明当前使用了什么临时脚本或人工检查替代。

## 手动验收

第一阶段实现完成后，至少手动验证以下路径：

- 默认语言跟随系统：中文系统显示中文，非中文系统显示英文。
- Settings 中可以切换 `中文`、`English`、`跟随系统`。
- 切换语言后，Settings 导航、菜单和常见按钮即时更新。
- macOS 顶部菜单在切换语言后完成重建并显示新语言。
- 终端命令输出不被本地翻译层改写。
- AI 模型回复不被本地翻译层改写。
- 服务端返回内容不被本地翻译层改写。
- 缺失中文 key 时，开发环境能暴露问题，发布环境能回退到英文。

## 分阶段落地建议

1. 新增 `crates/warp_i18n`，先接入 `en` 与 `zh-CN` 资源加载、`Locale`、`LanguagePreference`、`t!` 和受限的 `tr`。
2. 新增 key parity 与裸字符串检查入口，先覆盖新迁移的文件，避免一次性要求全仓库清零。
3. 接入 Settings 语言偏好，完成持久化、系统语言检测和运行时通知。
4. 迁移 Settings 导航、通用按钮、菜单和启动登录路径。
5. 增加命令面板高频项翻译，并明确哪些命令标题仍使用英文专有名词。
6. 补齐 macOS 菜单重建逻辑和跨平台手动验收记录。

每一阶段都应保持英文资源完整可用。中文翻译可以逐步扩展，但不能破坏英文默认体验。

## 非目标

本计划不包含：

- 绕过官方服务端能力、订阅、账号、同步或 AI 限制。
- 修改许可证、移除上游署名或隐藏 fork 来源。
- 对 Warp 官方品牌作重新授权声明。
- 翻译终端、AI、服务端和用户内容。
- 一次性完成全仓库所有 UI 字符串迁移。
